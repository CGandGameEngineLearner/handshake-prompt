# Threat Model

This document describes the security assumptions, threat scenarios, and
mitigations for the Handshake Prompt Protocol.

## Assumptions

1. **The service uses HTTPS in production.** All HPP traffic, including
   WebSocket upgrades, is expected to run over TLS. The protocol does not
   add transport encryption on top.
2. **The browser is the trust anchor.** The user has already logged into
   the service in their browser (via cookie, OAuth, SSO, etc.) when they
   click the "AI Action" button. HPP grants the Agent a session derived
   from that browser-level identity.
3. **The Agent runs on the user's own machine** (Codemaker, Claude
   Desktop, Cursor, etc.) and is trusted not to actively betray the user.
   HPP does not protect against a malicious Agent running on the user's
   own system — at that point the user has bigger problems.

## In scope

- Confidentiality and integrity of session tokens
- Replay and brute-force resistance
- Protection of user-entered data from accidental AI overwrite
- Bounded blast radius for leaked credentials

## Out of scope

- TLS/transport security (rely on HTTPS)
- Browser malware that can read clipboard and DOM
- Compromised Agent code

## Tokens & Session IDs

| Value | Entropy | Generated by |
|-------|---------|--------------|
| `sessionId` | 128 bits | `secrets.token_hex(16)` |
| `token`     | 192 bits | `secrets.token_urlsafe(24)` |

**Threat: brute-force token guessing.**
Mitigation: 192 bits ≈ 6×10⁵⁷ possible values, far beyond practical guessing.
Combined with rate limiting (60 req/min/session) and 30-min TTL, an attacker
gets at most ~1,800 guesses against a single sid before it expires.

**Threat: timing side-channel during token comparison.**
Mitigation: `secrets.compare_digest` (constant-time) in both Python and JS
implementations.

**Threat: enumerating valid sids.**
Mitigation: 128-bit sids are also unguessable. Requests against unknown
sids return a uniform 404 without revealing internal state.

---

## Token lifetime

Default TTL is **30 minutes from last access** (sliding window). Tokens
expire automatically; no server-side revocation API is exposed because the
expected lifetime is shorter than a typical revocation latency anyway.

**Threat: leaked prompt persists indefinitely.**
Mitigation: short TTL bounds the damage to one session. If the user
forwards the prompt by accident, the worst case is "someone might fill the
same form within the next 30 minutes."

**Threat: leaked prompt allows _watching_ the user fill the form via WS.**
Yes, this is possible during the TTL window. Treat handshake prompts as
sensitive — the UX should make this clear in the copied text ("Do not
forward this prompt").

---

## Rate limiting

Default: **60 requests / minute / session**, sliding window.

**Threat: AI loops or buggy code rapidly hammering the server.**
Mitigation: per-session limit + 429 responses make this self-correcting.

**Threat: distributed brute-force.**
Mitigation: 192-bit token entropy makes per-sid attack infeasible. Cross-sid
brute force is bounded by the cleanup of expired sessions.

---

## User-data protection

The single most important behavioral rule of HPP:

> **The server refuses to let an AI overwrite a field whose `by` is
> `user` or `user_edit`.**

This is enforced inside `Session.set_value` in the server SDK and cannot
be bypassed by clients. AI submissions that target a user-written field
are added to the `rejected` list and **not applied**.

**Threat: AI accidentally clobbers a careful manual entry.**
Mitigation: protocol-level lock; covered by tests in the SDK.

**Threat: malicious Agent intentionally clobbers.**
Mitigation: same lock. Even a malicious Agent can't write through it.

---

## WebSocket security

WebSocket connections to `/ws/handshake/<sid>` require a `?token=<token>`
query parameter, verified at upgrade time with `secrets.compare_digest`.
Connections without a valid token are closed immediately.

**Threat: cross-origin WebSocket hijack.**
Mitigation: tokens are required and unguessable. Even if a malicious page
guesses a session ID, it cannot connect without the token.

**Note:** tokens in URLs may appear in server access logs. Operators
should configure logging accordingly (drop the `token` query param from
access logs) or accept this as a known trade-off (WS spec does not allow
custom headers in the browser).

---

## Browser-side endpoints

`/handshake/notify/<sid>` does not require a token; it accepts user-edit
notifications from the browser. By default, this endpoint trusts any
caller on the same origin (CORS + same-site cookies provide the boundary).

Operators with stricter requirements can pass a
`require_browser_auth=callable` to `HandshakeManager`, which is invoked
with `(request, session)` and must return True to accept the call.

**Threat: an attacker forges a userEdit to mark a field as
`by=user_edit` and block the AI from filling it.**
Mitigation: the only effect is **denial of service against the AI**, not
data exfiltration or corruption. If this matters in your context, enable
`require_browser_auth`.

---

## In-memory state

The reference Python SDK stores sessions in-memory by default. A process
restart loses all in-flight sessions. This is acceptable for the protocol's
design (sessions are short-lived anyway).

For multi-instance deployments, supply a custom `SessionStore` (e.g.
Redis-backed) when constructing `HandshakeManager`.

---

## Recommendations for operators

1. **Always serve over HTTPS in production.** No exceptions.
2. **Configure server access logs to redact `token` query params.**
3. **Consider shortening TTL** for highly sensitive use cases (e.g. 5 min).
4. **Display the copied prompt** to the user before they paste it, so they
   know what they're handing over.
5. **Audit the `applied` / `rejected` lists** in `on_done` hooks for
   anomaly detection.
